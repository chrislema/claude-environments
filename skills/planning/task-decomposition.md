# Skill: Task Decomposition

## Use When

- turning product documents into implementation work
- refining a broad request
- repairing a weak plan

## Guidance

- prefer concrete deliverables over vague work themes
- make hard dependencies explicit
- keep task boundaries coherent and reviewable
- include acceptance criteria that can be checked
- escalate only ambiguity that genuinely blocks planning

## Output Expectations

Each task should include:

- task id
- owner role
- deliverable
- dependencies
- acceptance criteria
- owned files or surfaces when possible
